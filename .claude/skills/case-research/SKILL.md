# 事例リサーチ & 検証レポート作成

お題から「検証済み」のクリエイティブ事例リサーチレポート（`<テーマ>_RESEARCH.md`）を作る。
**検証されていない事例をレポートに載せることが最大の事故**（Cannes2026監査で受賞レベル/部門誤り13件が発覚した実績がある）。速さより検証を優先する。

**実行体制**: メインは指揮と最終判断のみ。収集= `case-collector`、機械検証= `link-checker`、受賞照合= `award-verifier`、執筆= `report-writer` の各エージェントに委任する（モデルは各エージェント定義に固定済み。このスキルでモデルを指定しない）。一括実行は Workflow `case-research-sweep`（args にお題）でも可。

## Phase 0: お題の確定

ユーザーのお題から以下を確定する。不明な点は推測で埋めて「Xと仮定して進めます」と明記（質問は最終手段）:
- **テーマ / 用途**（何の企画のインスピレーション源か）
- **調査範囲**: 期間（デフォルト: 直近10年）/ 地域（デフォルト: 国内外）/ ジャンル横断の可否
- **切り口の粒度**: 「手法別」に分類するのが基本（GPS連動 / 参加型 / ARG型…のように）

## Phase 1: ローカル資産を先に引く（必須・最速）

Web検索の前に必ず `data/cases.json`（検証済み455件+）を検索する:

```bash
node scripts/search-cases.mjs <キーワード…> --format md --limit 15
node scripts/search-cases.mjs --tag Tech/AI --tag Form/Website --year 2023-2026 --format md
```

- ここでヒットした事例は**検証済み扱いでそのままレポートに使える**（受賞情報・リンク・概要とも監査通過済み）
- タグ語彙は `data/tag-vocabulary.json`（Tech/Form/Theme の3軸）
- 過去レポート（リポジトリ直下の `*_RESEARCH.md`）に同テーマがないかも `grep -l` で確認し、重複調査を避ける

## Phase 2: Web リサーチ（多角ファンアウト）

`case-collector` エージェントを**検索角度ごとに並列で**起動し、候補リストだけ受け取る（メインで検索を回さない）:
- 角度の例: 賞レース（Cannes/One Show/D&AD/ADFEST）、業界メディア（AdAge/Contagious/宣伝会議/AdverTimes）、手法名（"scavenger hunt campaign" 等）、プラットフォーム名
- 各エージェントには「お題・角度・英日両言語で引くこと」を指示する

## Phase 3: 事実検証（ここを省略しない）

レポートに載せる**全事例**を検証エージェントに通す（検証基準の詳細は各エージェント定義が正）:

1. **リンク死活・oEmbedタイトル照合** → `link-checker` に一括委任
2. **受賞主張の一次情報照合** → 受賞主張のある事例だけ `award-verifier` に委任
3. **検証できなかった項目**は削除するか `⚠️未検証:` を明記して残す。無言で載せない。`award-verifier` が「誤り」と返したものは訂正値を使う

## Phase 4: レポート生成

執筆は `report-writer` エージェントに委任する（検証済み材料と以下の形式を渡す）。既存の `ALBUM_SITE_RESEARCH.md` の構成に合わせる:

```markdown
# <テーマ> リサーチレポート
## ——<サブタイトル（何のための事例集か）>

**調査範囲**: <期間> / <地域> / <ジャンル>
**対象**: <用途>
**検証**: 受賞情報・リンクは <日付> 時点で確認済み

## 目次
（切り口ごとのアンカーリンク）

## 1. <切り口名>
### ◎ <事例名>「<キャンペーン名>」 (<年>)
**コア体験**: <1行。この事例の本質>
**URL**: <検証済みURL>
**仕組み**: （箇条書き）
**受賞**: <検証済みの正確な賞・部門・レベル>（該当時のみ）
**<お題>への応用ポイント**: <転用のヒント>
```

- 保存先: リポジトリ直下 `<テーマ英語スネーク大文字>_RESEARCH.md`（既存の命名に合わせる）
- 末尾に「調査ソース一覧」と「未検証・要確認リスト」を付ける

## Phase 5: 事後オプション（提案のみ、自動実行しない）

- ResearchMan 本体への追加候補があれば「radar 候補」として事例リスト（title/client/year/link/videoId）を提示する。cases.json への追加はユーザーが指示した場合のみ（追加時は id 規則・サムネイル・pre-push 監査があるため OPERATIONS.md §6 に従う）
- プレゼン用が必要なら `/research-deck` でHTMLデッキ化できると案内する
